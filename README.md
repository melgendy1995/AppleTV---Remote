# Apple TV Remote — native Swift app (macOS + iOS)

A SwiftUI app for controlling Apple TVs from a Mac, iPhone, or iPad. It drives a
small **FastAPI + [pyatv](https://github.com/postlund/pyatv)** backend that does
the actual device discovery, pairing, and protocol work — pyatv's protocol
implementations are too involved to reimplement in Swift, so the Python backend
stays. On macOS the app **bundles and launches that backend automatically**; on
iOS you point it at a backend running on a Mac on the same LAN.

## How it fits together

```
┌──────────────────────┐     REST + WebSocket      ┌───────────────────┐   pyatv    ┌──────────┐
│  SwiftUI app          │  ───────────────────────▶ │  FastAPI backend   │  ───────▶ │ Apple TV  │
│  (macOS / iOS)        │  ◀─────────────────────── │  (uvicorn + pyatv) │  ◀─────── │  (tvOS)   │
└──────────────────────┘   /api/* , /ws/state        └───────────────────┘            └──────────┘
```

- **macOS:** `BackendController` finds the bundled Python in
  `…/Contents/Resources/Vendor/python`, picks a free port, spawns
  `python -m uvicorn app.main:app`, waits for `/api/status` to answer, and kills
  the subprocess on quit. If `Vendor/` isn't bundled (plain dev builds), it falls
  back to whatever backend URL is set in Settings.
- **iOS:** there's no bundled backend; set the backend URL in Settings to the
  Mac running it (e.g. `http://192.168.1.20:8000`).

### Backend (`Vendor/backend/app/`)

| File             | Responsibility |
| ---------------- | -------------- |
| `main.py`        | FastAPI app — REST endpoints + `/ws/state` WebSocket, CORS, optional static frontend mount |
| `atv_manager.py` | Wraps pyatv: scan, connect, send commands, now-playing, keyboard, push/poll listeners |
| `pairing.py`     | Session-based pairing that walks each pairable protocol (AirPlay, Companion, MRP, RAOP) collecting PINs |
| `storage.py`     | Persists pairing credentials to `~/.appletv-remote/credentials.json` (chmod 600) |
| `models.py`      | Pydantic models for the API responses |

## Build & run

### 1. Generate the Xcode project

```bash
brew install xcodegen        # one-time
cd swift-app
xcodegen generate            # re-run whenever project.yml changes
open AppleTVRemote.xcodeproj
```

### 2. Build / run

In Xcode pick a scheme (`AppleTVRemote-macOS` or `AppleTVRemote-iOS`), choose a
destination, and Cmd+R.

From the CLI:

```bash
# macOS
xcodebuild -project AppleTVRemote.xcodeproj -scheme AppleTVRemote-macOS \
  -destination 'platform=macOS' -derivedDataPath build \
  CODE_SIGN_IDENTITY="-" CODE_SIGNING_REQUIRED=NO build
open "build/Build/Products/Debug/Apple TV Remote.app"

# iOS Simulator
xcodebuild -project AppleTVRemote.xcodeproj -scheme AppleTVRemote-iOS \
  -destination 'platform=iOS Simulator,name=iPhone 15' build
```

### 3. Running the backend yourself (dev / iOS)

The macOS app starts the bundled backend on its own. To run it manually — for
iOS, or for a dev build without `Vendor/python` — install the deps and launch
uvicorn:

```bash
cd Vendor/backend
pip install -r requirements.txt        # fastapi, uvicorn, pyatv, python-multipart
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Then set the backend URL from the gear icon in the app. Default is
`http://localhost:8000`; it's stored in `@AppStorage("backendURL")`.

## Features

- **Device discovery** — scans the LAN for Apple TVs and merges in saved devices.
- **In-app pairing** — session-based PIN flow that walks each pairable protocol
  (AirPlay, Companion, MRP, RAOP) and saves credentials per protocol.
- **Connect / disconnect / forget** saved devices.
- **Remote controls** — D-pad (up/down/left/right/select), Menu, Home (+ hold),
  play/pause/stop, next/previous, skip, volume, screensaver, Control Center.
- **Live now-playing** — title/artist/album/app, position, volume, and artwork,
  streamed over a WebSocket (`/ws/state`) with auto-reconnect.
- **Keyboard** — a modal pops automatically when a tvOS text field is focused;
  supports set / append / clear and pushes text to the TV. Focus is detected via
  a `text_get()` poll because pyatv's cached focus state can go stale.
- **Settings** — editable backend URL with a test-connection check.
- macOS: `Cmd+,` opens Settings; the bundled backend's stderr tail is surfaced
  if it fails to start.

## HTTP / WebSocket API

The Swift client (`BackendClient`) and backend (`main.py`) speak this surface:

| Method | Path | Purpose |
| ------ | ---- | ------- |
| GET    | `/api/status` | connection state + supported commands |
| GET    | `/api/devices?timeout=N` | scan for devices |
| GET    | `/api/devices/saved` | saved devices + credentials |
| DELETE | `/api/devices/{identifier}` | forget a device |
| POST   | `/api/connect/{identifier}` | connect using saved credentials |
| POST   | `/api/disconnect` | disconnect |
| POST   | `/api/pair/start` | begin pairing `{identifier}` |
| POST   | `/api/pair/{session_id}/pin` | submit a PIN `{pin}` |
| POST   | `/api/pair/{session_id}/cancel` | cancel a pairing session |
| POST   | `/api/command/{command}` | send a remote command (+ optional args) |
| GET    | `/api/playing` | current now-playing state |
| GET    | `/api/artwork` | current artwork bytes |
| GET/POST | `/api/keyboard`, `/api/keyboard/append`, `/api/keyboard/clear` | keyboard text |
| WS     | `/ws/state` | streamed `playing` and `keyboard` events |

## Requirements

- macOS 14+ / iOS 17+, Swift 5.10, Xcode with XcodeGen.
- Backend: Python 3.12, `fastapi`, `uvicorn[standard]`, `pyatv>=0.15`,
  `python-multipart`.

## Known gaps

- No code-signing / notarization configured — builds run unsigned for local dev.
- Haptics on iOS for D-pad presses are not wired up.
- Shipping to the App Store needs a `DEVELOPMENT_TEAM` in `project.yml`, a
  re-run of `xcodegen generate`, and a decision about how the App Store build
  reaches a backend (bundling a pyatv helper makes it non-distributable through
  the App Store, so the realistic path is letting users enter a URL).
